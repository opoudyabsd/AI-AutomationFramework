**File:** `src/tests/e2e/graph-settings.spec.ts` · **Line:** 251 · **Check:** A2 · **Rule:** CLAUDE.md §3 ("Never place test logic inside pages/ or helpers" — converse: direct locator calls belong in the POM)

**Violation:**
```typescript
await expect(page.locator('.dcg-label')).toBeVisible();
```

**Fix:**
```typescript
// 1. Add to CalculatorPage constructor:
readonly complexExpressionLabel: Locator;
// ...
this.complexExpressionLabel = page.locator(SELECTORS.COMPLEX_LABEL);

// 2. Add to constants.ts SELECTORS:
```

## Retrospective

> Thematic root-cause analysis of the findings above.

---

### Theme 1 — Raw Page Access Bypasses the POM Contract (A2, C1, C3, J1)

**Observed pattern:** Four findings share the same root: the developer discovered a new DOM element (`.dcg-label`, the XPath grandparent traversal in `traceCoordinates`) during live debugging and used the raw selector inline in the spec and POM rather than registering it through the project's conventions.

**Root cause:** Debugging sessions create pressure to find the first working selector quickly. When `.dcg-trace-coordinates` disappeared and complex points had no Export button, the fastest path was `page.locator('.dcg-label')` inline and `locator('..')` chains — both working but both bypassing project conventions.

**How to prevent:**
- Treat every newly discovered selector as a `constants.ts` candidate *before* writing the assertion. The cost is one extra line; the benefit is centralized documentation and a single change point.
- When a project-wide rule must be violated (XPath grandparent, because no alternative exists today), add a `// KNOWN EXCEPTION:` comment immediately and open a tracking item rather than leaving silent non-conformance.
- Add a pre-commit lint rule or code review checklist item: "No `page.locator()` calls in `*.spec.ts` files."

---

### Theme 2 — Proxy Assertion Used as Full Verification (H4, K3 × 2)

**Observed pattern:** Two tests (TC-E2-E2E-002 and TC-E2-E2E-004) use available DOM proxies (aria-checked, `.dcg-label` visibility) but stop short of verifying the downstream effect the test case actually documents (period change, coordinate content).

**Root cause:** Canvas assertion limitations are correctly identified and documented, but the conclusion drawn is "skip the assertion entirely" rather than "find the closest achievable DOM proxy." This leaves tests that verify UI control state rather than application behaviour.

**How to prevent:**
- When a TC step cannot be verified directly, brainstorm the *closest available DOM signal* before omitting the step entirely. For period change: a hover near the radian peak and absence of a trace tooltip confirms the peak moved. For complex coordinate: `.dcg-label` text content asserts which point is labelled.
- Add a `// COVERAGE GAP:` comment with a description of what cannot be asserted, so reviewers know the omission is intentional and tracked.
- During TC authoring, mark steps that are automation-challenging with an explicit "Automation proxy: …" note (TC-E2-E2E-002 already does this for step 7 but not steps 2 and 11).

---

### Theme 3 — Test Name Wording Drift from TC (K5 × 4)

**Observed pattern:** All four spec test names diverge from their TC titles, with the TC-E2-E2E-002 divergence understating the assertion scope ("verifying expression renders" instead of "verify period change").

**Root cause:** Test names are often written after implementation, when the developer knows what was actually asserted rather than what should be asserted. This creates a gap between spec intent and automated coverage description.

**How to prevent:**
- Write the test name first, copying the TC title, then adjust only for the `should <verb>` format.
- When the automation materially deviates from the TC (e.g., skipping hover steps), update both the test name AND add an inline `// TC deviation:` comment so the gap is visible in the test report and code history.
